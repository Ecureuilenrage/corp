### Ce qui a bien fonctionne

- Le contrat Ticket est complet: create, update, move, cancel, run (foreground + background), board, artifacts.
- La discipline architecturale `validation -> event -> snapshot -> projection -> rendu` a ete tenue sur les 11 stories. C'etait l'action 3 de la retro Epic 1, et elle a ete pleinement respectee.
- La regle vendor-decoupled a ete rigoureusement respectee: aucun identifiant OpenAI/Codex n'a fuite dans le schema coeur. Tout est reste confine dans `executionHandle.adapterState`.
- La suite de tests cumulative a grandi de 41 a 137 tests, conformement a l'action 4 de la retro Epic 1.
- La strategie d'isolation dual (worktree Git + copie workspace) fonctionne, avec gestion Windows.
- La decision de lancer une revue adversariale apres les 5 premieres stories a ete un tournant: elle a evite de partir en Epic 3 avec des fondations viciees.
- Les 5 projections JSON sont completes et fonctionnelles: mission-status, ticket-board, resume-view, artifact-index, approval-queue.

### Ce qui n'a pas bien fonctionne

- **190 findings sur 5 stories**: ratio de defauts anormal. Plus de la moitie de l'epic (6/11 stories) a ete consacree a des corrections de bugs non planifiees.
- **Bugs critiques silencieux empiles sur plusieurs stories**: `resume-view.json` jamais ecrit (C1 depuis 2.1), `ticket.in_progress` jamais emis en foreground (C6 depuis 2.3), perte silencieuse d'artefacts en exception (C12 depuis 2.5).
- **Reversions de design entre stories**: mission `failed` acceptant/refusant les nouveaux tickets (2.10 vs 2.11, B-16), `structural-compare` triant des tableaux ordonnes (2.10 vs 2.11, B-06).
- **Statuts fantomes**: `CLOSED_OPEN_TICKET_STATUSES` contenait `"completed"` et `"closed"`, des valeurs inexistantes dans le type `TicketStatus` (B-08). L'agent d'implementation a hallucine des statuts.
- **Tests incomplets dans 2.11**: plusieurs subtaches de tests restent non cochees (ticket.retried event, guards T-01 a T-04, adapter tests T-05/T-06, cancel/update edge cases, cross-projection consistency, test fragilities TF-26 a TF-31).

### Causes racines identifiees

1. **Specs insuffisantes sur les invariants defensifs**: les acceptance criteria des stories 2.1-2.5 decrivaient le happy path mais pas les guards, cas d'erreur ni invariants a proteger.
2. **L'agent d'implementation ne challenge pas les specs**: Codex implemente fidelement les specs y compris leurs lacunes. Quand une indication manque, il invente au lieu de signaler l'absence.
3. **Pas de feedback loop entre stories**: les 5 stories se sont empilees sans verification intermediaire, permettant aux bugs de se composer.

### Lecons apprises

- Les specs sont le levier numero un pour reduire les defauts en aval. Des acceptance criteria sans guards produisent des defauts en cascade.
- Le pipeline multi-agents fonctionne mais chaque maillon a ses angles morts. La revue adversariale est indispensable pour rattraper les lacunes cumulees.
- Une revue par epic est viable si les specs couvrent les cas defensifs et qu'un check mid-epic est planifie sur les stories critiques.
- La discipline architecturale paie: le patron journal -> event -> projection a tenu et les corrections ont solidifie les fondations.

## 2. Preparation de l'epic suivant
